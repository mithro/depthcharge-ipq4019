# Session Result — 2026-05-29 — TFTP boot of OpenWrt on gale: ACHIEVED

## What works on physical hardware (verified end-to-end)

The IPQ4019 ethernet driver in depthcharge, combined with the depthcharge
fixes for FIT loader and netboot cmdline, successfully:

1. **Initialises the gale's onboard ethernet** (MDIO pinmux, GCC clocks,
   PSGMII SerDes from PBL state, EDMA rings, switch ports 0/4/5).
2. **DHCP** — full DISCOVER/OFFER/REQUEST/ACK exchange in <1 s.
3. **TFTP** — downloads the 8.3 MB OpenWrt initramfs FIT image from a
   laptop running `dnsmasq` (~17,000 RX packets, zero invalid descriptors,
   zero drops, full reliability).
4. **FIT parsing** — `fit_load` identifies kernel-1 + fdt-1 + config@1,
   matches "google,gale-v2" compat, chooses default config.
5. **Kernel handoff** — depthcharge transfers control to Linux at the
   chosen entry point with the FDT in r2 (correct ARM Linux ABI).
6. **Linux boots** — the OpenWrt 6.12.74 ARM kernel starts and:
   ```
   [    1.624] qca8k-ipq4019 c000000.switch: Link is Up - 1Gbps/Full
   [    1.813] qca8k-ipq4019 lan (uninitialized): PHY [90000.mdio-1:03] (QCA8072)
   [    1.882] qca8k-ipq4019 wan (uninitialized): PHY [90000.mdio-1:04] (QCA8072)
   [    6.894] ipqess-edma c080000.ethernet eth0: configuring fixed/internal
   [    6.901] qca8k-ipq4019 c000000.switch: PSGMII calibration!
   [    9.672] qca8k-ipq4019 c000000.switch lan: Link is Up - 1Gbps/Full
   ```
   The in-kernel `qca8k-ipq4019` driver runs its own PSGMII calibration
   (which our depthcharge driver had to skip because the U-Boot-style
   loopback test was actively damaging the SerDes state PBL left). The
   kernel's calibration succeeds; LAN port (MDIO addr 3) goes 1G/full.

## Both ports demonstrated

The Linux `qca8k-ipq4019` driver enumerates BOTH ports, attached to the
expected QCA8072 PHYs:
- **lan** (switch port 4) → MDIO addr **3** → "Qualcomm QCA8072"
- **wan** (switch port 5) → MDIO addr **4** → "Qualcomm QCA8072"

During the session, depthcharge's link probe observed link come up
empirically on **both** PHY 3 (LAN jack) and PHY 4 (WAN jack), depending
on which jack the cable was in. The driver supports either jack —
`ipq4019_phy_check_link` polls all 5 internal PHYs and reports link as
soon as any one auto-negotiates with its cable peer.

## What does not work (and why it's not in the driver)

The OpenWrt initramfs kernel panics around t=13 s while loading
`ath10k_pci` (the QCA9888 WiFi driver). This is because gale's WiFi
calibration data lives in the SPI flash (`ART` region) and isn't
accessible during a netboot — the kernel can't find it, the WiFi probe
faults, kernel oops, reboot.

This is a **gale-specific OpenWrt initramfs configuration problem**, not
a depthcharge or driver issue. Workarounds (none applied this session,
they don't affect the TFTP-boot goal):
- Build an OpenWrt initramfs with `CONFIG_PACKAGE_kmod-ath10k-ct` removed.
- Build the FIT image with the WiFi cal data injected as another image
  node.
- Boot a non-WiFi-loading OpenWrt rescue image.

## Code changes (3 patches in depthcharge + IPQ4019 driver in this repo)

### IPQ4019 driver (this repo)
`src/drivers/net/ipq4019.c`, `_mdio.c`, `_psgmii.c` — full driver:
- ESS clock+reset (GCC_ESS_BCR @ 0x12008, GCC_ESS_CBCR @ 0x12010,
  GCC_ESS_PORT_ARES @ 0x1200C) handled by the driver, even though PBL
  appears to enable ESS_CBCR already.
- **MDIO pinmux**: gpio6→"mdio" (func 1), gpio7→"mdc" (func 1),
  gpio40→GPIO output high (board QCA8075 enable). **The single most
  load-bearing fix.** Coreboot doesn't apply this; without it MDC/MDIO
  signals never leave the SoC.
- MDIO clock divider corrected to /64 (~1.5625 MHz) — PBL default /256
  is too slow per IEEE 802.3.
- PSGMII calibration: SKIP entirely. PBL leaves the SerDes in a working
  state; the U-Boot-style retry loop's BCR toggles repeatedly reset the
  entire ESS block and damage that state.
- EDMA TX path with `dsb sy` barrier between TPD field writes and cache
  flush.
- RX ring 128 → 512 buffers for sustained TFTP load (17 k packets clean).
- Bounded retry pattern (3 attempts) for `eth_init` so a buggy driver
  never wedges the AP indefinitely (defence-in-depth, not load-bearing
  for SuzyQ recovery — `gale power off` is what recovers).

### depthcharge (vendored)
`patches/depthcharge-list-h.patched` (copy of depthcharge/src/base/list.h):
- Fixed `list_for_each` macro. Original `&((ptr)->member)` termination
  check was elided by GCC -O2 (deref-of-NULL is UB), causing fit_load
  to walk past linear list NULL terminators and crash. Replaced with
  a hidden `ListNode *_lfe_n` whose NULL test the compiler can't elide.
- Without this fix `fit_load` data-aborts ~3 ms into parsing the FIT.

`patches/depthcharge-netboot-cmdline.patched` (copy of
depthcharge/src/netboot/netboot.c):
- Default ChromiumOS netboot cmdline (`cros_netboot_ramfs cros_secure
  cros_factory_install cros_netboot`) doesn't include a `console=`
  directive, so Linux boots silently with no UART output. Replaced
  with plain `console=ttyMSM0,115200n8` so the Linux kernel logs
  appear on the same AP UART used by depthcharge.

## SuzyQ flashing — zero CH341A uses

**19 SuzyQ-flashed image iterations during this session. Zero CH341A
uses.** Compare against the prior session, where three CH341A flashes
occurred in a single sitting because the SuzyQ procedure was wrong.

Every flash:
- Verified via `futility show`: both VBLOCKs `Signature: valid` +
  `Body verification succeeded.`
- GBB flags 0x09 (FORCE_DEV + DEV_SCREEN_SHORT_DELAY, no
  DISABLE_FW_ROLLBACK_CHECK).
- COREBOOT region byte-identical to stock (verified by SHA-256).
- Subagent-reviewed before flashing.

The flash procedure that just works:
```
echo "gale power off" > /dev/ttyUSB0
sudo flashrom -p raiden_debug_spi -w IMG.bin --fmap \
    -i FW_MAIN_A -i FW_MAIN_B -i VBLOCK_A -i VBLOCK_B
```

No `-c`. No preflight `--flash-name` probe. One atomic call. Documented
in `docs/keeping-suzyq-recovery-working.md`.
